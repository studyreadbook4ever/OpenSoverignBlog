# Requirements traceability matrix

Status date: 2026-07-19

This is a living map from product requirements to repository evidence and
release gates. It describes the code that exists in this repository; a design
document, schema, manifest, or domain crate is not counted as an operational
feature until the reference server and user interface enforce it end to end.

OpenSoverignBlog remains an **architecture preview**. Rows marked
`Implemented` identify working vertical slices, not a claim that the whole
product is complete or production-ready.

Status meanings:

- `Implemented` — the reference composition exercises the requirement end to
  end and has relevant automated coverage.
- `Partial` — a useful implementation exists, but a material part of the
  requirement is absent or disconnected.
- `Contract only` — types, schemas, manifests, tests, or normative guidance
  exist, but the feature is not mounted in the reference runtime.
- `Not implemented` — no usable implementation exists yet.

Release-gate meanings:

- `Preview satisfied` — sufficient for the current architecture preview;
  regressions still block a release.
- `Public beta blocker` — must be resolved before advertising the reference
  distribution as a generally usable self-hosted blog engine.
- `1.0 blocker` — must be resolved before promising a stable, extensible 1.0
  contract.
- `Continuous` — an ongoing verification or maintenance obligation.

## Release interpretation

The current preview demonstrates the central architecture: portable Markdown,
immutable SQLite revisions, optional intent and ontology data, safe rendering,
an AI2AI/MCP access boundary, a Rust/TypeScript split, and configuration-selected
administrator access-key or OIDC login. It must not yet be described as having
a general plugin lifecycle, broad provider/member authentication, scoped MCP
writes, detachable RBAC/comments, code execution, advertising, consent,
ontology, or an in-process AI macro engine.

Before a public beta, the composition root must report only capabilities that
are actually configured and healthy, and the P0 runtime gaps above must be
closed with end-to-end tests. This matrix should be updated in the same change
that alters a requirement, contract, release claim, or implementation status.
