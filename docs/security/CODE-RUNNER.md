# Code Runner Security Model

- Status: Normative for any production code-runner implementation
- Applies to: code-runner client, runner service, execution images, and
  artifact transport
- Last updated: 2026-07-18

## Purpose

The code runner executes user- or model-supplied code. Such code is hostile by
default. The runner is an optional subsystem and is not part of the trusted
publishing server.

This document defines the minimum boundary for enabling code execution. If a
deployment cannot provide these controls, code execution must remain disabled.
Stored source and previous artifacts remain readable when the runner is
disabled.

The words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY describe requirement
strength.

## Security goals

The runner must:

- prevent a job from reading or modifying the publishing database, content
  blobs, plugin packages, configuration, or secrets;
- prevent one job from observing or affecting another;
- bound CPU, memory, process count, filesystem use, wall time, and output;
- deny network access unless an administrator explicitly grants a narrow
  destination policy;
- make every submission and result attributable and auditable;
- fail closed without making the publishing server unavailable;
- make stale, duplicate, forged, and replayed requests harmless.

## Non-goals and limits

The runner does not protect against an administrator with root access to the
host. The baseline also does not claim complete resistance to kernel
vulnerabilities, hardware side channels, denial of service against an
unpartitioned host, or malicious firmware.

A plain child process, language virtual environment, chroot by itself, or
container running with broad host privileges is not an adequate security
boundary. Higher-risk public execution SHOULD use a stronger isolation layer
such as a dedicated host or microVM in addition to the baseline controls.

## Threat actors and inputs

Potentially hostile inputs include:

- code pasted by an authenticated user;
- code or arguments generated by an AI agent;
- dependencies, package manifests, archives, and execution images;
- filenames, MIME types, environment-like values, and output metadata;
- a compromised code-runner plugin;
- forged or replayed jobs;
- malicious stdout, stderr, HTML, SVG, notebooks, and downloadable artifacts.

Authentication of the submitting user does not make submitted code trusted.
AI provenance does not make code trusted.

## Trust boundaries

The architecture is:

    Browser or AI workflow
              |
       Publishing server
       policy + job record
              |
       authenticated broker
              |
        isolated runner
              |
       ephemeral sandbox

The publishing server owns authorization, policy resolution, job metadata, and
the durable artifact record. The runner owns only the bounded execution
lifecycle.

The following rules are absolute:

- The publishing server MUST NOT evaluate submitted code.
- The publishing server MUST NOT mount a Docker, Podman, containerd, or other
  host runtime socket in order to start jobs.
- The runner MUST NOT mount the publishing SQLite file, blob store, plugin
  directory, configuration directory, secret store, or backup directory.
- The runner and publishing server SHOULD use different operating-system users
  and SHOULD be separately deployable.
- A runner compromise MUST NOT provide a credential that authorizes content,
  administrative, plugin-management, or secret-store APIs.

## Job request

Every accepted job has a server-generated immutable identifier and policy
snapshot. A request contains only the data needed to execute:

- job ID, attempt ID, creation time, and expiry time;
- submitting principal and site IDs for audit, not ambient authorization;
- a pinned execution profile or image digest;
- entrypoint and structured arguments;
- immutable input artifact digests;
- declared output types;
- resource limits;
- network policy;
- a single-use capability token scoped to this job and attempt;
- an idempotency key and request signature.

The runner MUST reject unknown fields when they could alter execution
semantics. It MUST reject expired jobs, reused attempt tokens, mismatched
digests, unsupported profiles, and limits outside administrator-defined
bounds.

Raw publishing sessions, API keys, model credentials, advertising credentials,
and database URLs MUST NOT appear in a job.

## Authorization and policy

Code execution requires both:

1. an authenticated principal allowed to execute the requested profile; and
2. the site policy permitting the requested language, limits, network policy,
   and artifact types.

A plugin's code.execute capability is only permission to request execution. It
does not bypass user authorization, site policy, review requirements, or
runner limits. AI-generated requests are subject to the same checks and SHOULD
require review unless a narrowly defined automation policy says otherwise.

The effective policy snapshot is stored with the job so later configuration
changes do not obscure why execution was allowed.

## Sandbox baseline

Each attempt MUST run in a fresh sandbox. A production baseline includes:

- an unprivileged, rootless execution identity;
- no-new-privileges;
- all unnecessary Linux capabilities dropped;
- a restrictive syscall profile;
- a read-only root filesystem;
- a fresh writable work directory with a hard quota;
- no host paths except purpose-built, read-only input staging;
- no device access;
- no privileged mode, host PID namespace, host IPC namespace, or host network;
- a bounded process and thread count;
- a bounded CPU allocation and CPU time;
- a hard memory limit with no access to host swap where practical;
- a wall-clock deadline enforced outside the sandbox;
- bounded file size, file count, log size, and total output;
- deterministic termination of the complete process tree;
- destruction of the writable sandbox after artifact collection.

The runtime MUST launch an argument vector directly and MUST NOT concatenate
untrusted strings into a shell command. The working directory, filenames,
archive paths, and entrypoint are validated before execution.

Images and language profiles are administrator-approved and pinned by digest.
Mutable tags such as latest MUST NOT identify production execution images.
Package installation during a job is disabled by default.

## Filesystem and archive handling

Input files are staged by digest into a new job directory. The staging
implementation MUST defend against:

- absolute paths and parent traversal;
- symbolic-link and hard-link escapes;
- case-folding and Unicode filename collisions;
- device files, sockets, and named pipes;
- sparse-file and decompression bombs;
- archive entries that overwrite previous entries;
- time-of-check/time-of-use replacement.

Input staging is read-only to the sandbox. Output collection walks only the
dedicated output directory without following links. It enforces count and byte
limits before importing data into the artifact store.

The runner MUST NOT trust a filename extension or process-supplied MIME type.
The publishing server serves risky artifacts as downloads from an isolated
origin where possible. HTML and SVG are never inserted into the admin or
published page without the renderer's separate sanitization and trust policy.

## Network policy

Network access is disabled by default.

When a profile permits network access:

- destinations are allowlisted by scheme, normalized hostname, port, and
  optionally path;
- loopback, link-local, private, multicast, metadata-service, and Unix-socket
  destinations are denied unless a dedicated local service is explicitly part
  of the profile;
- DNS resolution and connection enforcement occur at a trusted egress boundary
  to resist rebinding and alternate-address attacks;
- redirects are re-evaluated against the same policy;
- request and response sizes, connection count, rate, and duration are bounded;
- raw packet access and listening sockets remain disabled;
- the audit record identifies the allowed policy and observed destination.

A hostname allowlist implemented only inside hostile code is not enforcement.
For reproducible builds, dependencies SHOULD be prefetched into a reviewed
execution image or brokered through a controlled package mirror.

## Secrets

The default execution policy exposes no secrets.

If a future execution profile genuinely requires a credential, it must use a
separate runner-scoped secret with minimum privilege, not a publishing-server
or user credential. The credential is bound to one profile or destination,
short-lived where supported, redacted from logs, and omitted from artifacts.

The preferred design is a capability broker that performs the authorized
operation without disclosing the underlying secret. Environment-variable
injection of general server secrets is prohibited.

## Result and artifact handling

The runner reports a structured terminal state:

- succeeded;
- failed;
- timed out;
- resource limit exceeded;
- cancelled;
- policy rejected;
- runner lost.

Stdout and stderr are untrusted byte streams. They are byte-limited, decoded
with replacement where necessary, have terminal control sequences removed for
display, and are never interpreted as HTML. Truncation is explicit.

Each output artifact records:

- digest and size;
- detected media type;
- producing job and attempt;
- execution profile and image digest;
- timestamps;
- optional content scanner result;
- whether collection was complete or truncated.

The publishing server verifies reported digests before accepting an artifact.
A successful process exit does not automatically publish or execute its
output. AI-generated output becomes a proposed artifact or draft revision and
passes the normal review policy.

## Service authentication and replay defense

Server-to-runner communication MUST be authenticated. A single-host deployment
may use a permission-restricted local socket; a network deployment uses
mutually authenticated transport or an equivalently strong service identity.

Capability tokens are:

- randomly generated by the server;
- scoped to one job, attempt, operation, and expiry;
- unusable as publishing API credentials;
- consumed or recorded on first successful use;
- excluded from normal logs and error messages.

Result submission is idempotent. A late result from an earlier attempt cannot
replace the accepted result of a newer attempt.

## Lifecycle and cleanup

The durable job state machine is:

    queued -> leased -> preparing -> running -> collecting -> terminal

Leases expire. Retrying creates a new attempt and sandbox while preserving the
job ID. Cancellation revokes outstanding capabilities, terminates the whole
process tree, and schedules cleanup.

Cleanup is idempotent and runs after success, failure, timeout, cancellation,
and runner restart. A periodic reconciler removes orphaned sandboxes and
temporary artifacts. Cleanup failure is observable and blocks reuse of the
affected workspace.

## Availability and failure behavior

The runner is optional. If it is unavailable:

- content reads, edits, and unrelated publication remain available;
- new execution requests remain queued or fail according to explicit policy;
- publication that requires a fresh code artifact is blocked or uses a
  previously approved artifact according to the document's recorded policy;
- the server does not fall back to local execution;
- repeated crashes can open a circuit breaker and disable the runner feature.

Runner health is not equivalent to permission to execute. Readiness includes
the ability to create the configured sandbox and enforce its limits.

## Logging, audit, and privacy

Audit events include submission, authorization decision, effective policy
digest, lease, start, network grant, cancellation, terminal state, and artifact
acceptance. They include identifiers and limits but not source content,
secrets, tokens, or full stdout by default.

Operational logs use structured fields and redact authorization headers,
cookies, query credentials, and values marked sensitive. Retention for source,
logs, and artifacts is independently configurable. Deletion of an artifact
does not rewrite the historical fact that a job produced it.

## Required release tests

The following tests are release gates for enabling the runner:

### Isolation

- A job cannot read the publishing SQLite file or blob directory.
- A job cannot inspect host processes, devices, runtime sockets, or another
  job's workspace.
- Parent traversal, absolute paths, symlinks, hard links, and archive escapes
  are rejected.
- The complete descendant process tree is killed on cancellation and timeout.
- The writable filesystem is destroyed after every terminal state.

### Resource enforcement

- CPU-bound, memory-bound, process-forking, disk-filling, log-flooding, and
  never-ending jobs hit their configured limits.
- Output count, single-file size, total size, stdout, and stderr limits are
  enforced before durable import.
- Runner restart reconciles a job without accepting duplicate terminal
  results.

### Network

- Network is absent in the default profile.
- Loopback, private, link-local, metadata, IPv6 alternate forms, encoded host
  forms, DNS rebinding, and redirect escapes are denied.
- An explicitly allowed destination works only on its allowed scheme and port.

### Authentication and policy

- Expired, forged, replayed, wrong-job, and wrong-attempt tokens fail closed.
- A plugin capability without user or site permission cannot execute.
- A user allowed in one site cannot execute against another site's job.
- Limits above administrator maxima are rejected, not silently expanded.

### Artifact safety

- Hostile HTML, SVG, terminal escapes, invalid UTF-8, deceptive MIME types,
  decompression bombs, and oversized results do not become executable admin
  content.
- Artifact digest mismatch rejects the result.
- A late result cannot overwrite a newer accepted attempt.

### Failure behavior

- Runner absence never causes in-server fallback.
- Timeout, cancellation, sandbox creation failure, output collection failure,
  and cleanup failure yield distinct terminal or recoverable states.
- Circuit breaking leaves the rest of the publishing engine healthy.

Tests SHOULD run against the actual production sandbox configuration in CI or
a dedicated security test environment. Mock-only tests are insufficient.

## Development mode

A local development mode MAY use weaker isolation only when:

- it is explicitly named unsafe;
- it binds only to a developer-controlled interface;
- it cannot be enabled by a content document or plugin;
- production startup rejects the setting unless a separate operator override
  is present;
- the UI displays a persistent warning;
- tests never treat it as evidence that production isolation works.

Development convenience must never create an automatic production fallback.

## Incident response

Operators must be able to:

- disable new execution immediately;
- revoke server-to-runner credentials;
- identify jobs, profiles, images, and artifacts affected by a runner version;
- retain or quarantine audit records without serving hostile artifacts;
- rotate runner-scoped credentials;
- remove affected images by digest;
- restart publishing safely with the runner feature disabled.

Evidence of sandbox escape is treated as host compromise. The runner host is
removed from service and rebuilt; restarting only the job is not sufficient.
